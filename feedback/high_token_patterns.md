# 고토큰 소비 원인 분석

이 문서는 Claude가 작업 중 컨텍스트(토큰)를 많이 소비한 원인과 방지책을 기록한다.
다음 대화에서 같은 실수를 반복하지 않는 것이 목적이다.

---

## 1. 대용량 파일 직접 읽기

**원인**: 아래 파일들을 Read 도구로 열면 파일 전체가 컨텍스트에 올라간다.

| 파일 | 크기 | 컨텍스트 영향 |
|------|------|------------|
| `data/korean-dict/kr_korean.csv` | 9.7MB / 508,142줄 | 수만 토큰 즉시 소진 |
| `data/korean-dict/kr_korean_simple.csv` | 5.5MB / 508,142줄 | 수만 토큰 즉시 소진 |
| `step3/bigram_stats.txt` | 172,685줄 | 대량 토큰 소진 |
| `step4/trigram_stats.txt` | 659,449줄 | 대량 토큰 소진 |

**방지책**: 위 파일은 절대 Read 도구로 열지 않는다. 확인 필요 시 `head -N`, `grep`, `wc -l` 만 사용.

## 2. 세그폴트 디버깅 왕복 (step4 개발 중)

**원인**: `load_trigram`에서 600,000개짜리 배열 3개를 스택에 선언 → 세그폴트 → 원인 파악 → 파일 읽기 → 수정 → 재컴파일 → 재실행으로 여러 라운드 소비.

```c
// 잘못된 코드 (스택 오버플로우)
uint64_t ctx_keys[600000];  // ~4.8MB
int      ctx_cnt[600000];   // ~2.4MB
long     ctx_sum[600000];   // ~4.8MB  → 합계 ~12MB, 스택 한계 초과
```

**방지책**: 단일 배열이 `항목 수 × sizeof(타입) > 100,000 바이트`이면 반드시 `malloc`.

## 3. 컴파일 에러로 인한 왕복 (step4 개발 중)

**원인**: step3 코드를 복붙한 후 구조체 필드명(`TransEntry.to_cp` → `TransEntry.cp`) 불일치를 사전에 확인하지 않아 컴파일 에러 → 파일 읽기 → 수정 → 재컴파일 왕복.

**방지책**: 이전 step 코드를 복붙할 때 구조체 필드명을 먼저 grep으로 확인 후 진행.
```bash
grep -n "TransEntry\|\.cp\b\|\.to_cp" step3/step03_bigram_sample.c
```

---

## 4. 결과 파일 파싱 버그로 인한 왕복 (report 작성 중)

**원인**: Python에서 `len(word) // 3`으로 한글 글자 수를 계산 → 실제값과 다른 결과 → 재실행. Python `str`의 `len()`은 바이트가 아니라 유니코드 문자 수이므로 `// 3` 불필요.

**방지책**: Python에서 한글 처리 시 `len(word)`가 곧 글자 수임을 기억.

---

## 5. 새 step 작성 전 확인 체크리스트

이전 step을 n-gram 차수만 올려 확장할 때 미리 확인하면 디버깅 왕복을 줄일 수 있다.

| 항목 | 확인 방법 |
|------|---------|
| 키 타입 오버플로우 여부 | `TOTAL^n` 계산 후 uint32_t 한계(4.3×10⁹) 초과 시 uint64_t |
| 대형 배열 스택 할당 여부 | `항목 수 × sizeof > 100,000` → `malloc` |
| 파일 형식 변경 범위 | START 토큰 수 = n-1, 파서 수정 전 형식 먼저 정의 |
| 구조체 필드명 일치 여부 | `grep` 으로 사전 확인 |
